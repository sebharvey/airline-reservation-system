# Schedule Microservice â€” API Specification

> **Service owner:** Schedule domain
> **Base path:** `/v1`
> **Transport:** HTTPS (TLS 1.2 minimum)
> **Content type:** `application/json`

The Schedule microservice is the internal persistence layer for flight schedule definitions. It accepts a full season schedule payload, replaces all existing schedule records atomically, and returns a per-schedule import summary. Schedules are identified by a unique `ScheduleId` UUID generated at import time.

> **Important:** The Schedule microservice is an internal service. It is not called directly by channels (Web, App, NDC). All requests are routed through the **Operations API** orchestration layer, which handles authentication before forwarding calls to this service. See the [Security](#security) section for details on how calls are authenticated.

---

## Security

### Authentication

The Schedule microservice sits behind the Operations API orchestration layer. Ops Admin App users authenticate via the Operations API using OAuth 2.0 / OIDC:

1. The ops user obtains a JWT access token by authenticating against the Identity microservice via the Operations API.
2. The ops user sends the access token as a `Bearer` token in the `Authorization` header on all subsequent Operations API requests.
3. The Operations API validates the JWT signature using the Identity microservice's public signing key (RS256 or ES256) â€” no database round-trip required.
4. Once validated, the Operations API forwards the request to the Schedule microservice.

Calls from the Operations API to the Schedule microservice are authenticated using an **Azure Function Host Key**, passed in the `x-functions-key` HTTP header. The Schedule microservice itself does not validate JWTs; that responsibility belongs to the Operations API. See [Microservice Authentication â€” Host Keys](../api.md#microservice-authentication--host-keys) in `api.md` for the full mechanism, key storage, and retrieval details.

### Required Headers

| Header | Required | Description |
|--------|----------|-------------|
| `Content-Type` | Yes | Must be `application/json` |
| `Authorization` | Yes (at Operations API layer) | `Bearer {accessToken}` â€” JWT with 15-minute TTL, validated by the Operations API before the request reaches the Schedule MS |
| `x-functions-key` | Yes (on all Operations API â†’ Schedule MS calls) | Azure Function Host Key authenticating the Operations API as an authorised caller. See [`api.md` â€” Microservice Authentication](../api.md#microservice-authentication--host-keys) for key storage and retrieval details |
| `X-Correlation-ID` | Yes | UUID generated at the channel boundary; propagated on every downstream call for distributed tracing and log correlation |

### Data Protection

- Schedule data is not customer-facing PII. Standard logging practices apply.
- The `createdBy` field records the identity reference of the ops user who created the schedules â€” this should not be exposed to external channels.

## Endpoints

### POST /v1/schedules

Replaces all existing flight schedule records with the supplied season schedule payload. The operation is atomic: all existing records are deleted and the new set is inserted within a single transaction. Returns a summary of the imported schedules.

**When to use:** Called by the Operations API after parsing a season schedule (from SSIM or a pre-built JSON payload). The entire `[schedule].[FlightSchedule]` table is replaced by each call to this endpoint.

#### Request

```json
{
  "header": {
    "standard": "IATA",
    "airlineCode": "AX",
    "seasonStart": "2026-01-01",
    "seasonEnd": "2026-12-31",
    "fileType": "SCHED",
    "fileCreationDate": "2026-03-26"
  },
  "carriers": [
    {
      "airlineCode": "AX",
      "seasonCode": "01W",
      "validFrom": "2026-01-01",
      "validTo": "2026-12-31",
      "schedules": [
        {
          "flightNumber": "AX001",
          "origin": "LHR",
          "destination": "JFK",
          "departureTime": "08:00",
          "arrivalTime": "11:10",
          "arrivalDayOffset": 0,
          "daysOfWeek": 127,
          "aircraftType": "A351",
          "validFrom": "2026-01-01",
          "validTo": "2026-12-31",
          "createdBy": "schedule-import"
        }
      ]
    }
  ],
  "recordCount": 1
}
```

##### Top-level fields

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `header` | object | No | Metadata describing the source file; informational only â€” not persisted |
| `header.standard` | string | No | Standard identifier, e.g. `"IATA"` |
| `header.airlineCode` | string | No | Airline IATA designator |
| `header.seasonStart` | string (date) | No | Season start date, ISO 8601 |
| `header.seasonEnd` | string (date) | No | Season end date, ISO 8601 |
| `header.fileType` | string | No | File type descriptor |
| `header.fileCreationDate` | string (date) | No | Date the file was created |
| `carriers` | array | Yes | One or more carrier schedule groups; must contain at least one entry |
| `carriers[].airlineCode` | string | No | Airline IATA designator for this carrier group |
| `carriers[].seasonCode` | string | No | IATA season code, e.g. `"01W"` |
| `carriers[].validFrom` | string (date) | No | Season validity start for this carrier |
| `carriers[].validTo` | string (date) | No | Season validity end for this carrier |
| `carriers[].schedules` | array | Yes | Array of schedule definitions; must contain at least one entry |
| `recordCount` | integer | No | Expected total record count; informational only â€” not validated |

##### Schedule entry fields (`carriers[].schedules[]`)

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `flightNumber` | string | Yes | Flight number, e.g. `"AX001"`. Max 10 characters |
| `origin` | string | Yes | IATA departure airport code. Exactly 3 characters |
| `destination` | string | Yes | IATA arrival airport code. Exactly 3 characters |
| `departureTime` | string (time) | Yes | Local departure time at origin, `HH:mm` format |
| `arrivalTime` | string (time) | Yes | Local arrival time at destination, `HH:mm` format |
| `arrivalDayOffset` | integer | No | `0` = same day (default), `1` = next calendar day at destination |
| `daysOfWeek` | integer | Yes | Operating days bitmask: Mon=1, Tue=2, Wed=4, Thu=8, Fri=16, Sat=32, Sun=64. Range: 1â€“127 |
| `aircraftType` | string | Yes | IATA aircraft type code. Max 4 characters, e.g. `"A351"` |
| `validFrom` | string (date) | Yes | First operating date (inclusive), ISO 8601 |
| `validTo` | string (date) | Yes | Last operating date (inclusive), ISO 8601. Must not be before `validFrom` |
| `createdBy` | string | Yes | Identity reference of the ops user submitting the import. Max 100 characters |

> **Note:** `_comment` fields present in pre-built JSON payloads (e.g. `res/schedules/flight-schedule-2026-payloads.json`) are silently ignored by the deserialiser.

#### Response â€” `200 OK`

```json
{
  "imported": 48,
  "deleted": 45,
  "schedules": [
    {
      "scheduleId": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
      "flightNumber": "AX001",
      "origin": "LHR",
      "destination": "JFK",
      "validFrom": "2026-01-01",
      "validTo": "2026-12-31",
      "operatingDateCount": 365
    }
  ]
}
```

| Field | Type | Description |
|-------|------|-------------|
| `imported` | integer | Number of schedule records inserted |
| `deleted` | integer | Number of previous records removed before the import |
| `schedules` | array | Per-schedule summary for each imported record |
| `schedules[].scheduleId` | string (UUID) | Unique identifier generated for this schedule record |
| `schedules[].flightNumber` | string | Flight number |
| `schedules[].origin` | string | IATA departure airport code |
| `schedules[].destination` | string | IATA arrival airport code |
| `schedules[].validFrom` | string (date) | First operating date |
| `schedules[].validTo` | string (date) | Last operating date |
| `schedules[].operatingDateCount` | integer | Number of dates within `validFrom`â€“`validTo` that match the `daysOfWeek` bitmask |

#### Error Responses

| Status | Reason |
|--------|--------|
| `400 Bad Request` | Missing required fields, invalid field format (e.g. malformed IATA code, invalid bitmask, `validFrom` after `validTo`), or empty `carriers` / `schedules` arrays |

### GET /v1/schedules

Returns all persisted `FlightSchedule` records. Called by the Operations API during the inventory import flow (`POST /v1/schedules/import-inventory`) to retrieve the full set of stored schedules before generating `FlightInventory` records in the Offer MS.

**When to use:** Called by the Operations API to enumerate all schedules stored in the Schedule domain. Typical caller sequence: (1) import schedules via `POST /v1/schedules`, (2) retrieve them via `GET /v1/schedules`, (3) generate inventory via the Offer MS.

#### Request

No request body. No query parameters.

#### Response â€” `200 OK`

```json
{
  "count": 2,
  "schedules": [
    {
      "scheduleId": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
      "flightNumber": "AX001",
      "origin": "LHR",
      "destination": "JFK",
      "departureTime": "08:00",
      "arrivalTime": "11:10",
      "arrivalDayOffset": 0,
      "daysOfWeek": 127,
      "aircraftType": "A351",
      "validFrom": "2026-01-01",
      "validTo": "2026-12-31",
      "flightsCreated": 0,
      "operatingDateCount": 365
    }
  ]
}
```

| Field | Type | Description |
|-------|------|-------------|
| `count` | integer | Total number of schedule records returned |
| `schedules` | array | All persisted schedule records |
| `schedules[].scheduleId` | string (UUID) | Unique identifier for the schedule |
| `schedules[].flightNumber` | string | Flight number, e.g. `"AX001"` |
| `schedules[].origin` | string | IATA departure airport code |
| `schedules[].destination` | string | IATA arrival airport code |
| `schedules[].departureTime` | string (time) | Local departure time at origin, `HH:mm` |
| `schedules[].arrivalTime` | string (time) | Local arrival time at destination, `HH:mm` |
| `schedules[].arrivalDayOffset` | integer | `0` = same day, `1` = next day |
| `schedules[].daysOfWeek` | integer | Operating days bitmask (Mon=1 â€¦ Sun=64) |
| `schedules[].aircraftType` | string | IATA aircraft type code |
| `schedules[].validFrom` | string (date) | First operating date (inclusive), ISO 8601 |
| `schedules[].validTo` | string (date) | Last operating date (inclusive), ISO 8601 |
| `schedules[].flightsCreated` | integer | Number of `FlightInventory` records created from this schedule (0 if not yet imported to inventory) |
| `schedules[].operatingDateCount` | integer | Number of dates within `validFrom`â€“`validTo` that match the `daysOfWeek` bitmask |

#### Error Responses

| Status | Reason |
|--------|--------|
| `500 Internal Server Error` | Database error |
