# Data Storage Principles
**Apex Air — Reservation Platform**

---

## Data Ownership

- **Each microservice is the single owner of its data.**
  No service may read from or write directly to another service's database schema. Data ownership is enforced at the schema level (e.g. `offer.*`, `order.*`, `payment.*`) and, in a fully isolated deployment, at the physical database instance level. Cross-domain data access must always be mediated through a service's API or via event consumption.

- **Domain boundaries must be respected in schema design.**
  Cross-schema foreign key constraints must not be declared in the shared database instance. Where a reference to another domain's entity is required (e.g. `InventoryId` in `delivery.FlightManifest`), the reference is stored as a plain column with integrity maintained at the application layer by the orchestration layer, which controls write sequencing.

- **Denormalisation is permitted within a service's own schema for operational convenience, with clear documentation.**
  For example, `FlightNumber` and `DepartureDate` are denormalised onto `delivery.FlightManifest` to avoid cross-service joins at query time. Such denormalisation must be explicitly noted and the canonical source of truth identified.

---

## Schema Design

- **Database schemas must use domain names as the schema prefix, matching the service's capability domain name.**
  This applies uniformly: `offer.*`, `order.*`, `payment.*`, `delivery.*`, `customer.*`, `identity.*`, `seat.*`. Schema names must be used consistently across all code, migrations, and documentation.

- **Primary keys must use `UNIQUEIDENTIFIER` (UUID v4) generated by the database (`NEWID()`) unless a compelling reason for sequential keys exists.**
  UUIDs avoid sequential enumeration of entity IDs in APIs, reduce PK collision risk in distributed writes, and support future migration to physically separated database instances without key conflicts.

- **All tables must include `CreatedAt` and `UpdatedAt` timestamp columns using `DATETIME2` in UTC.**
  `SYSUTCDATETIME()` must be used as the default to ensure UTC storage regardless of server locale. `UpdatedAt` must be maintained by the application layer or a trigger on every write.

- **Immutable tables (audit and transaction logs) must be append-only and must not expose update or delete paths through any service API.**
  Examples include `payment.PaymentEvent` and `customer.LoyaltyTransaction`. These tables provide the audit trail and source of truth for running balances; their integrity depends on immutability being enforced at both the database and application layers.

- **Check constraints must be used to enforce structural validity on JSON columns.**
  `NVARCHAR(MAX)` columns storing JSON documents (e.g. `OrderData`, `BasketData`, `CabinLayout`) must carry an `ISJSON` check constraint to prevent malformed documents being written. Structural schema validation beyond this must occur at the application layer before write.

- **Nullable columns must be used deliberately and documented.**
  Nullable columns signal optional data (e.g. `BookingReference` on `payment.Payment` before an order is confirmed, `TravelDocument` for passengers who haven't yet provided it). Columns that are nullable only transiently should be reviewed — a separate state machine or status column may be more appropriate.

## JSON Document Storage

- **JSON columns are appropriate for complex, variable-structure documents that are read and written as a unit.**
  `OrderData`, `BasketData`, and `CabinLayout` are suitable candidates because they are consumed holistically by services and their internal structure varies across bookings. They must not be used to avoid relational modelling for data that is regularly queried, filtered, or joined on individual fields.

- **Scalar fields used for indexed lookups, lifecycle management, routing, or eventing must be stored as typed columns, not only inside JSON.**
  For example, `OrderStatus`, `BookingReference`, `ChannelCode`, `TotalAmount`, and `ExpiresAt` are typed columns on their respective tables. This ensures query performance and index support. Fields present as typed columns must not be duplicated inside the JSON document.

- **SQL Server computed columns with JSON path expressions must be used to index frequently queried JSON properties.**
  Where filtering or sorting on a JSON property is operationally required, a persisted computed column extracting the value must be created and indexed, rather than relying on full-document scans.

- **JSON documents must be versioned to support schema evolution.**
  Documents should include a version indicator (e.g. `"schemaVersion": 1`) to allow consumers to apply version-appropriate parsing logic as the document structure evolves over time. Migration strategies for documents stored under previous schema versions must be defined before deploying breaking changes.

## Indexing Strategy

- **Indexes must be designed to support the read access patterns of the owning service and its known consumers.**
  Every table must have at minimum its primary key index. Additional indexes must be justified by a documented query pattern. Over-indexing is a performance liability; indexes must be reviewed and dropped if unused.

- **Partial (filtered) indexes must be used where queries consistently target a subset of rows.**
  For example, the basket expiry job queries only `Active` baskets; the corresponding index filters on `BasketStatus = 'Active'`. This reduces index size and improves scan performance for the targeted workload.

- **Unique indexes must be used to enforce business-level uniqueness constraints.**
  Examples include `BookingReference` on `order.Order`, `LoyaltyNumber` on `customer.Customer`, `PaymentReference` on `payment.Payment`, and the compound seat/passenger uniqueness constraints on `delivery.FlightManifest`. These constraints are part of the domain's invariants and must not be left solely to application-layer enforcement.

## Transient vs Persistent Data

- **Basket data is transient pre-sale state and must be hard-deleted on successful order confirmation.**
  The confirmed `OrderData` JSON is the authoritative post-sale record. The basket serves no purpose once an order exists and must not be retained as a shadow copy. Expired and abandoned baskets must be retained briefly (e.g. 7 days) for diagnostics before automated purge.

- **Stored offers are point-in-time pricing snapshots and must have a defined expiry.**
  `offer.StoredOffer` rows must carry an `ExpiresAt` timestamp. A background job must periodically purge or archive expired, unconsumed offers to prevent the table growing unboundedly. The Order API must validate offer expiry before consumption.

- **Refresh tokens are the only authentication artefact persisted to the Identity DB.**
  Short-lived JWT access tokens must not be stored. Refresh tokens must be stored as a hash only, with the raw token returned to the client once and discarded server-side after hashing.

- **Session-scoped identifiers (e.g. `SeatOfferId`) must not be stored long-term by channels or services.**
  These identifiers are valid only for the duration of a booking session and must not be persisted in client-side storage beyond the session.

## Data Lifecycle and Retention

- **Data retention periods must be defined per domain, reflecting both regulatory requirements and data minimisation obligations.**
  Financial and manifest records must be retained for a minimum of 7 years in line with airline regulatory requirements. Personal data must be purged or anonymised promptly once the retention period lapses.

- **Automated purge jobs must enforce retention policies without manual intervention.**
  Jobs must be idempotent, observable (emitting metrics and alerts on failure), and must log each purge action with the count and type of records affected. Purge logic must be tested in staging before production deployment.

- **A data inventory must be maintained documenting what personal and financial data is held, where, and for how long.**
  This inventory is required to support UK GDPR compliance, data subject rights requests, and regulatory audit. It must be updated whenever a new service, table, or JSON field storing PII or financial data is introduced.

## Data Quality and Integrity

- **Running balance columns (e.g. `PointsBalance`, `TierProgressPoints`) must be updated atomically with their corresponding transaction row.**
  The `BalanceAfter` snapshot on each `LoyaltyTransaction` row provides a self-consistent audit trail. In the event of a discrepancy between the balance column and transaction history, the `BalanceAfter` on the most recent transaction is the source of truth.

- **Referential integrity between domains must be maintained at the application/orchestration layer, not via cross-schema database foreign keys.**
  This is a deliberate consequence of domain isolation. The Retail API orchestration layer is responsible for ensuring that cross-domain references (e.g. `InventoryId` in `delivery.FlightManifest`) are valid at write time.

- **Database migrations must be backwards-compatible and applied without downtime.**
  Additive changes (new nullable columns, new tables, new indexes) must be preferred over destructive changes. Column renames and type changes must follow an expand-contract pattern: add the new column, migrate data, update consumers, then drop the old column in a separate release.

- **Data validation must occur as close to the point of creation as possible.**
  The Delivery microservice validates `SeatNumber` against the active seatmap before writing any `FlightManifest` row; the Seat microservice is called at write time, not only at the API boundary. This prevents invalid data propagating into the system and simplifies downstream trust assumptions.

## Database Technology

- **Microsoft SQL Server is the standard RDBMS for all transactional domains.**
  All microservices share a single SQL Server instance with logical separation by schema for this project. Future physical separation into individual database instances per domain must be architecturally supportable without requiring changes to service code, only to connection configuration.

- **Database configuration and schema must be managed as code, version-controlled, and applied via a migration pipeline.**
  Manual schema changes to production are prohibited. All DDL changes must be applied through a reviewed and automated migration process (e.g. Flyway, EF Core migrations, or equivalent). Migration scripts must be idempotent where possible.

- **Connection pooling must be configured and monitored for each service.**
  Azure Functions isolate process pools; connection pool settings must be tuned to the concurrency profile of each function. Pool exhaustion must generate an alert.

- **Read replicas or read-only endpoints must be used for reporting, analytics, and non-operational queries.**
  Operational queries (booking confirmation, payment, check-in) must target the primary write endpoint. Analytical or reporting workloads must not compete with transactional workloads on the primary.
